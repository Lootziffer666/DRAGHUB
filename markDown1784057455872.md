Containerisierte Projektumgebung





Onlook lädt Projekte in einen Webcontainer, startet sie, zeigt die Vorschau im iframe, indexiert den Code und verknüpft sichtbare DOM-Elemente mit ihren Quelldateien. 



Das ist sehr nah an der äußeren Gestalt von ANVIL.



Aber der harte Haken



Onlook unterstützt derzeit zuverlässig:



Next.js + Tailwind + DOM



Nicht:



Kotlin Compose

Godot

Unity

Unreal

Android-native Views



Nicht-Next.js- und Nicht-Tailwind-Projekte stehen im eigenen README noch ausdrücklich als offene Aufgaben. Die Architektur könne theoretisch auf andere deklarative DOM-Frameworks erweitert werden – der Schwerpunkt liegt momentan aber auf Next.js und Tailwind. 



ANVIL dagegen ist aktuell bewusst ein Kotlin-Multiplatform-/Compose-Multiplatform-System für Android und JVM Desktop. Seine aktive Entwicklung liegt ausschließlich in anvil-kmp/; der alte JS-Prototyp ist historisch.



Onlook könnte diese Compose-Oberfläche also nicht einfach öffnen und visuell bearbeiten.

Zwei mögliche Wege



Weg A — Onlook als Modul in ANVIL



ANVIL bleibt KMP. Webprojekte öffnen sich in einem eingebetteten Onlook-Modul.



Commander

├── Web Designer      → Onlook

├── Flow Contract     → React Flow/tldraw

├── Code              → gemeinsamer Editor

├── Preview           → projektspezifischer Adapter

└── Runs/Gates        → ANVIL Core



Vorteil: Bestehende ANVIL-Architektur bleibt erhalten.



Nachteil: Die Oberfläche besteht weiterhin teilweise aus Compose und teilweise aus eingebettetem Web.



Weg B — ANVILs Oberfläche wird eine Web-App



Onlook wird die Basis der neuen Commander-Oberfläche. ANVIL Core und Module laufen dahinter als lokale Dienste oder Prozesse.



Onlook-basierter Commander

          ↓

      ANVIL Core API

          ↓

 Kotlin · Python · Rust · Godot · Node · Docker



Vorteil: Ein echter universeller Editor ist im Web wesentlich leichter umzusetzen. Monaco, Terminals, Graphen, DOM-Previews, Diff-Viewer und eingebettete Tools existieren dort bereits.



Nachteil: Die aktuelle Compose-Commander-Arbeit würde teilweise zur Übergangslösung.



Meine klare Einschätzung



Du könntest nicht ganz ANVIL mit Onlook abbilden.



Aber du könntest mit Onlook vermutlich 60–70 % der sichtbaren ANVIL-Werkbank übernehmen und ANVILs deutlich wertvolleren Teil dahinter erhalten:



Orchestrierung



Verträge



Gates



Provenienz



sprachunabhängige Module



reproduzierbare Outputs





Und ehrlich: Für dein Ziel eines Editors für unterschiedlichste Tools ist eine Weboberfläche wahrscheinlich die bessere Plattform als Compose Multiplatform.
