Die richtige Architektur ist:



> Ein physisches Exemplar, beliebig viele logische Verwendungen.







Gemeinsamer Objektspeicher

        │

        ├── Repo A Session 1

        ├── Repo A Session 2

        ├── Repo B

        └── Repo C



Keine vollständigen Repo-Klone



Statt:



sessions/

├── mythic-1/ kompletter Clone

├── mythic-2/ kompletter Clone

└── mythic-3/ kompletter Clone



gibt es einen gemeinsamen Git-Objektspeicher:



objects/repos/MYTHIC.git



und Sessions sind nur Worktrees:



sessions/

├── mythic-main/

├── mythic-pr-42/

└── mythic-experiment/



Die Dateien werden logisch getrennt angezeigt, teilen sich aber dieselben Git-Objekte. Nur tatsächlich veränderte Dateien erzeugen zusätzlichen Speicher.

Dasselbe gilt für Abhängigkeiten



Nicht jedes Repo bekommt eigene physische Kopien von:



React

Next.js

Godot

ONNX Runtime

Python Wheels

Gradle Libraries

Rust Crates

Modelle



Stattdessen landen alle Inhalte in einem hashbasierten gemeinsamen Speicher:



objects/

├── sha256-a81f...  React-Paket

├── sha256-b552...  ONNX Runtime

├── sha256-f991...  Modellgewicht

└── sha256-31ae...  Godot-Binary



Jedes Projekt besitzt nur ein Manifest:



{

  "dependencies": [

    "sha256:a81f...",

    "sha256:b552...",

    "sha256:f991..."

  ]

}



Zwei Repos mit derselben Datei referenzieren dasselbe Objekt.



Entscheidender Unterschied



Im Projekt verfügbar: ja

Physisch im Projekt dupliziert: nein





---



Die lokale Session wird nur eine dünne Änderungsschicht



┌────────────────────────────┐

│ Schreibbare Session-Schicht│ ← nur deine Änderungen

├────────────────────────────┤

│ Repo-Dateien               │ ← geteilt, unverändert

├────────────────────────────┤

│ Dependencies               │ ← geteilter Objektspeicher

├────────────────────────────┤

│ Runtime / Toolchain        │ ← geteilte Container-Layer

└────────────────────────────┘



Das ist Copy-on-Write:



Lesen erfolgt aus der gemeinsamen Basis.



Erst bei einer Änderung entsteht eine private Kopie.



Beim Löschen der Session verschwindet nur diese dünne Änderungsschicht.



Gemeinsame Dateien bleiben genau einmal gespeichert.





Ein Repo mit 20 GB Assets benötigt dann nicht bei fünf Sessions 100 GB. Die fünf Sessions können zusammen nur wenige zusätzliche Megabyte belegen, solange sie nicht jeweils große Dateien verändern.





---



Selbst unterschiedliche Repos können dedupliziert werden



Nicht bloß verschiedene Sessions desselben Repos.



Wenn WIZARD und SHADED beide dasselbe 500-MB-Modell enthalten, wird anhand des Dateihashes erkannt:



WIZARD/model.onnx ───┐

                     ├── sha256:91bf... → eine Datei

SHADED/model.onnx ───┘



Das funktioniert unabhängig von:



Dateiname



Ordner



Repository



Programmiersprache



Branch





Gleicher Inhalt bedeutet gleiches Objekt.

Für eingecheckte Dependencies



Du sagtest, alle Dependencies sollen bereits im Repo liegen. Das kann weiterhin logisch gelten.



ANVIL importiert sie beim ersten Öffnen:



Repo enthält dependency.bin

        ↓

Hash berechnen

        ↓

Objekt bereits vorhanden?

   ├── ja → Repo-Datei durch Referenz ersetzen

   └── nein → einmal im Object Vault speichern



Im virtuellen Workspace erscheint die Datei weiter normal an ihrer ursprünglichen Stelle. Intern ist sie aber nur ein Link, Reflink oder gemountetes Objekt.



Der Nutzer und das gestartete Programm merken davon nichts.

Für ANVIL wäre das die richtige Speicherhierarchie



ANVIL_HOME/

├── objects/          # Jeder Inhalt nur einmal

├── git/              # Gemeinsame Git-Objekte

├── runtimes/         # OCI-/Toolchain-Layer

├── manifests/        # Welches Repo braucht was?

├── sessions/         # Nur Copy-on-Write-Deltas

└── persistent/       # Ausdrücklich zu behaltende Daten



Löschlogik



Session schließen



Prozesse beenden

temporäre Schreibschicht löschen



Änderungen sichern



Git-Commit oder Patch erzeugen

neue Objekte in den gemeinsamen Speicher übernehmen



Garbage Collection



Ein Objekt wird erst gelöscht, wenn es von keinem Repo, Commit, Build oder gespeicherten Artifact mehr referenziert wird.

Die passende Metapher



Nicht hundert Werkstätten, in denen jeweils dieselben Maschinen stehen.



Sondern:



> Ein zentrales Werkzeuglager, aus dem jede virtuelle Werkstatt genau die benötigten Werkzeuge eingeblendet bekommt.







Die Werkstätten sind getrennt. Die Werkzeuge sind nicht dupliziert.



Damit wird der lokale Codespace nicht zum Speicherfresser, sondern eher zu einem virtuellen Dateisystem für Repositories:



Git + Content Addressing + Copy-on-Write + Runner



Das ist der richtige Unterbau für ANVIL. Nicht „Repos klonen und behalten“, sondern Repos materialisieren, solange sie laufen.
