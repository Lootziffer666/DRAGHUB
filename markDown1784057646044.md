Stable — genau. Und der entscheidende Gedanke ist:



GitHub enthält nicht „Version 1, Version 2, Version 3“ als drei Projektkopien.

Git speichert unveränderliche Objekte und setzt daraus Zustände zusammen.



Dein GitHub-Browser sollte dieses ohnehin vorhandene Prinzip endlich sichtbar und benutzbar machen.



Basiszustand

    +

Änderung A

    +

Änderung B

    =

aktuelle Version



Nicht:



projekt-final/

projekt-final-neu/

projekt-final-neu-wirklich/



Das eigentliche Modell



Jede geöffnete Version besteht aus drei Schichten:



┌──────────────────────────────┐

│ aktuelle, ungesicherte Deltas│

├──────────────────────────────┤

│ ausgewählter Commit          │

├──────────────────────────────┤

│ unveränderte Git-Objekte     │

└──────────────────────────────┘



Ein Checkpoint erzeugt keinen neuen Komplettsatz von Dateien.



Er erzeugt im Wesentlichen:



neue Blobs nur für geänderte Inhalte,



einen neuen Tree für die betroffenen Pfade,



einen Commit, der auf den vorherigen Zustand verweist.





Unveränderte Inhalte werden weiter referenziert.



Damit werden Git-Begriffe plötzlich verständlich



Klassischer Git-Begriff	Im Browser



Commit	Checkpoint

Branch	Variante oder Arbeitsraum

Tag	benannte stabile Version

Diff	Änderungsschicht

Merge	Varianten zusammenführen

Revert	Zustand zurücksetzen

Worktree	parallel geöffnete Variante

Blob	unveränderliches Inhaltsobjekt





Du musst dem Nutzer nicht einmal ständig Git-Vokabular zeigen.



Er sieht:



Aktueller Zustand

├── OAuth ergänzt

├── Repo-Picker verändert

└── Settings noch ungesichert



Und darunter:



[Checkpoint erstellen]

[Variante abzweigen]

[Änderungen verwerfen]

[Mit main vergleichen]



Besonders stark wird das für KI-Arbeit



Heute behandelt ein Coding-Agent ein Repo häufig wie einen veränderbaren Ordner.



Dein Browser könnte stattdessen jeden Agentenlauf als Delta behandeln:



Commit A

   ↓

Agentenauftrag

   ↓

Änderungsschicht

   ↓

visuelle Prüfung

   ├── übernehmen → Commit B

   ├── teilweise übernehmen

   └── verwerfen



Damit bekommt ein Agent nicht einfach die Macht, „das Projekt umzubauen“.



Er liefert einen Vorschlag auf einer neuen Schicht.



Das passt direkt zu ANVILs Gates:



Plan

  ↓

temporäre Variante

  ↓

ausführen

  ↓

Diff + Preview + Flow prüfen

  ↓

Commit oder verwerfen



Versionierung direkt während der Bearbeitung



Der Browser könnte sogar automatisch Mikro-Checkpoints anlegen:



Prompt abgeschickt

→ temporärer Checkpoint



Agent fertig

→ Ergebnis-Checkpoint



manuelle Korrektur

→ lokaler Delta-Zustand



Nutzer bestätigt

→ permanenter Commit



Die temporären Punkte müssen nicht alle als sichtbare Commits auf main landen.



Sie können zunächst sein:



lokale Refs,



versteckte Arbeitsbranches,



Draft-Commits,



Patch-Sets,



oder Session-Checkpoints.





Erst eine bestätigte Version wird als sauberer GitHub-Commit beziehungsweise PR veröffentlicht.



Der GitHub-Browser wird dadurch mehr als ein Dateibrowser



Er wird ein Zustandsbrowser.



Nicht:



Welche Dateien liegen im Repo?



Sondern:



Wie ist dieser Zustand entstanden?

Was wurde seitdem verändert?

Welche Alternative existiert?

Welche Änderung stammt von welchem Agenten?

Welcher Zustand wurde tatsächlich ausgeführt?



Eine Datei könnte beispielsweise diese Herkunft zeigen:



src/auth/OAuthDialog.tsx



Basis:

Commit 91af2c



Änderungen:

1. Sonnet – Dialog angelegt

2. Nutzer – Text verändert

3. Codex – Fehlerbehandlung ergänzt

4. Flow-Validator – Route durch Overlay ersetzt



Damit wird Provenienz nicht als zusätzliche Dokumentation geführt. Sie fällt direkt aus der Versionsstruktur heraus.



Der elegante Teil: Versionen können zusammengesetzt werden



Ein kompletter Projektzustand könnte intern als Manifest erscheinen:



{

  "base": "commit:91af2c",

  "layers": [

    "patch:oauth-overlay",

    "patch:repo-refresh",

    "patch:settings-domain"

  ]

}



Der Browser materialisiert daraus die sichtbare Arbeitsversion.



Das erlaubt:



Änderungen einzeln an- und auszuschalten,



Agentenergebnisse isoliert zu prüfen,



Varianten ohne vollständige Kopien zu erzeugen,



einzelne Features zwischen Branches zu übernehmen,



kaputte Schichten zu entfernen,



einen Zustand exakt wiederherzustellen.





Im Git-Endzustand können diese Schichten normale Commits werden. Während der Bearbeitung müssen sie aber nicht sofort die Commit-Historie zumüllen.



Wichtige Grenze: Git dedupliziert nicht alles ideal



Für Quellcode funktioniert das hervorragend.



Bei großen Binärdateien gilt:



Modell 4 GB, ein Byte verändert

→ aus Git-Sicht möglicherweise neuer großer Blob



Auch Assets, Datenbanken, Archive und generierte Builds gehören deshalb nicht blind in normale Git-Historie.



Dafür braucht dein Browser ergänzend:



Git

→ Code, Konfiguration, Manifeste, kleine Assets



Git LFS oder Object Vault

→ Modelle, große Binärdateien, Asset-Pakete



Artifact Store

→ Builds, Exporte, temporäre Ergebnisse



Im Repo liegt dann etwa:



{

  "asset": "village-model",

  "size": 4281172331

}



Der eigentliche Inhalt liegt einmal im Objektspeicher.

Meine klare Einordnung



Das ist nicht einfach „inkrementelles Speichern“.



Es ist:



Git-native zustandsbasierte Entwicklung



Git macht das intern bereits. Dein Browser macht es erstmals zu einem verständlichen Bedienmodell für Menschen und Agenten.



Und dadurch könnten drei bislang getrennte Dinge identisch werden:



Version

Agentenlauf

ausführbarer Projektzustand



Ein Zustand wird erstellt, ausgeführt, geprüft und anschließend entweder verworfen oder zur Geschichte des Repositories erklärt. Das ist deutlich eleganter als heutige IDE-, Codespace- und Vibe-Coding-Workflows.
