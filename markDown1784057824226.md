Stable — das verschwindet, sobald GitHub nicht mehr dein Fundament ist



Die 25-MiB-Grenze ist keine Git-Grenze. Sie gilt bei GitHub für Dateien, die über den Browser hochgeladen werden. Beim normalen Git-Push warnt GitHub ab 50 MiB und blockiert einzelne Dateien über 100 MiB. Das sind Regeln von GitHubs Infrastruktur.

ob große Dateien überhaupt in Git landen





Git selbst hat kein 25-MiB-Problem



Lokal kannst du grundsätzlich auch eine mehrere Gigabyte große Datei committen.



Das Problem ist nicht:



> Git kann große Dateien nicht speichern.







Sondern:



> Git behandelt jede gespeicherte Dateiversion als Teil der Repository-Geschichte.







Bei Quelltext ist das hervorragend. Bei großen Binärdateien wird es hässlich.



texture-pack.zip  4 GB

Version 1         4 GB

Version 2         möglicherweise weitere 4 GB

Version 3         möglicherweise weitere 4 GB



Git komprimiert und dedupliziert intern, aber bei bereits komprimierten Archiven, Modellen, Videos oder vielen Binärformaten bringt das oft wenig. Außerdem muss ein vollständiger Clone grundsätzlich die erreichbare Historie mitdenken.



Was Git LFS wirklich ist



Git LFS ist nicht Bestandteil von GitHub und auch kein magisches Dateiformat.



Es ist ein offenes Zusatzsystem:



Git-Repository

└── kleine Pointer-Datei



LFS-Speicher

└── eigentliche große Datei



Im Commit steht beispielsweise nur:



version https://git-lfs.github.com/spec/v1

oid sha256:4cac1962...

size 84977953



Beim Checkout liest Git LFS diesen Pointer und holt das dazugehörige Objekt aus einem separaten Speicher. 



GitHub betreibt lediglich einen LFS-Server dafür und setzt eigene Grenzen:



GitHub-Tarif	Maximale einzelne LFS-Datei



Free / Pro	2 GB

Team	4 GB

Enterprise Cloud	5 GB





Dazu kommen Speicher- und Transferabrechnung beziehungsweise Kontingente. 



Im Homelab



Du könntest Git LFS selbst hosten und deine eigenen Grenzen setzen.



Aber dein geplantes System kann es besser als klassisches LFS machen.



Dein Object Vault ist eigentlich „LFS richtig gedacht“



Statt:



Git

+ nachträglich angeflanschtes LFS

+ undurchsichtige Quoten

+ zusätzliche Befehle

+ Pointer-Probleme



baust du:



Git              → Code, Manifeste, Zustände

Object Vault     → große Inhalte

Workspace        → setzt beides transparent zusammen



Das Repo enthält dann:



{

  "path": "assets/models/castle.glb",

  "object": "sha256:91a4e9...",

  "size": 3847291451,

  "mediaType": "model/gltf-binary"

}



Im virtuellen Workspace erscheint trotzdem:



assets/models/castle.glb



Die Anwendung merkt nicht, dass die Datei aus einem gemeinsamen Objektspeicher eingeblendet wurde.

Noch besser: Chunk-Deduplizierung



Ein normaler LFS-Server speichert üblicherweise jede Dateiversion als separates vollständiges Objekt.



Du kannst große Dateien zusätzlich in feste oder inhaltsabhängige Blöcke zerlegen:



Datei Version 1

[A][B][C][D][E]



Datei Version 2

[A][B][X][D][E]



Physisch neu gespeichert wird nur:



[X]



Das Manifest der zweiten Version verweist weiterhin auf die vorhandenen Blöcke:



{

  "chunks": [

    "sha256:A",

    "sha256:B",

    "sha256:X",

    "sha256:D",

    "sha256:E"

  ]

}



Damit erhältst du inkrementelle Versionierung großer Binärdateien, bei denen Git allein und klassisches LFS schwach sind.



Aber eine wichtige Einschränkung



Chunk-Deduplizierung hilft nur, wenn kleine Änderungen nicht den gesamten restlichen Dateiinhalt verschieben oder verschlüsseln.



Gut geeignet:



große unkomprimierte Datenbestände



Disk-Images



bestimmte Modelle



Asset-Sammlungen



große strukturierte Binärdateien





Schlechter geeignet:



ZIP-Archive



verschlüsselte Dateien



manche komprimierten Videos



Formate, die bei einer kleinen Änderung die gesamte Datei neu schreiben





Darum sollte dein System Archive möglichst entpackt als Objektstruktur behandeln, statt ständig neue riesige ZIP-Dateien zu versionieren.

Die klare Architektur



Dein System

├── Git Object Database

│   └── kleine und diffbare Inhalte

├── Content-Addressed Object Vault

│   └── große und geteilte Inhalte

├── Artifact Store

│   └── reproduzierbare Ergebnisse

└── GitHub Adapter

    ├── Git Push

    ├── LFS bei Bedarf

    └── Release Uploads



Die 25-MiB-Grenze verschwindet vollständig.

Das LFS-Konzept bleibt sinnvoll, aber GitHubs konkrete LFS-Implementierung wird optional.



Dein System kann aus Git LFS die richtige Grundidee übernehmen – Pointer im Zustand, Inhalt separat – und den ganzen sichtbaren Verwaltungsballast eliminieren.
