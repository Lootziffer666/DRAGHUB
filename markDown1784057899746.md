Stable — genau so solltest du es bauen



Handy = Auftrag und Entscheidung.



Rechner = Sandbox und Ausführung.



NAS = langsames Archiv, nicht Arbeitslaufwerk.



CUE = unveränderliches Journal darüber, was wann warum passiert ist.



Die WLAN-Geschwindigkeit wird dadurch fast irrelevant, weil das Handy keine Repositories und großen Dateien verschiebt. Es sendet nur kleine Befehle.



Handy

  │  Prompt, Auswahl, Freigabe, Textänderung

  ▼

ANVIL auf deinem Rechner

  │

  ├── lokaler temporärer Git-Worktree

  ├── Agenten und Runner

  ├── Preview, Diff, CUE-Prüfung

  └── lokale schnelle Object Database

             │

             │ langsame, sequentielle Hintergrund-Queue

             ▼

            NAS





Der zentrale Architekturentscheid



Das NAS darf nicht der aktive Workspace sein



Nicht:



Handy → NAS-Repo → Rechner arbeitet über WLAN darauf





Das wäre langsam, fehleranfällig und bei vielen kleinen Dateien besonders unangenehm.



Richtig:



Handy → Rechner → lokale Sandbox → NAS repliziert später





Der Rechner hat den autoritativen Arbeitszustand. Das NAS bekommt bestätigte Zustände und Objekte gemütlich nachgereicht.







Was du am Handy machst



Auf dem Handy öffnest du eine schlanke ANVIL-Weboberfläche oder PWA auf deinem Rechner.



Du kannst dort:





Repo auswählen



Prompt schreiben



Live Contract Preview korrigieren



Agent starten



Logs in Kurzform sehen



Vorschau öffnen



Diff prüfen



einzelne Änderungen an- oder abwählen



Commit benennen



Zustand akzeptieren oder verwerfen



CUE-Evidence ansehen







Das Handy überträgt dabei meistens nur wenige Kilobyte:



{

  "command": "apply_change",

  "repository": "MYTHIC",

  "baseCommit": "91af2c",

  "intent": "OAuth als Overlay statt neuer Seite",

  "constraints": {

    "allowNewRoute": false,

    "preserveRepoPicker": true

  }

}





Keine node_modules, keine Modelle, keine Assets, keine Repo-Kopie.







Was auf dem Rechner geschieht



Beim Öffnen eines Projekts erzeugt ANVIL einen lokalen Worktree:



ANVIL_HOME/

├── git/

│   └── MYTHIC.git             gemeinsamer Git-Objektspeicher

├── sessions/

│   └── MYTHIC/run-0184/       dünner Worktree

├── objects/                   große Inhalte einmalig

├── queue/                     ausstehende Schreibaufträge

└── evidence/                  CUE-Nachweise





Der Ablauf:



1. Commit auswählen

2. Worktree materialisieren

3. Änderung lokal ausführen

4. Anwendung starten

5. Diff und Evidence erzeugen

6. Nutzer entscheidet am Handy

7. lokaler Commit entsteht

8. Commit und Objekte wandern in die NAS-Queue





Der NAS-Upload beginnt nicht während jeder einzelnen Tastatureingabe.







Zwei getrennte Queues



Du brauchst tatsächlich zwei Warteschlangen.



1. Command Queue



Sie enthält Aufträge vom Handy:



001 Repo MYTHIC öffnen

002 OAuth-Flow ändern

003 Preview erzeugen

004 Diff akzeptieren

005 Commit veröffentlichen





Diese Befehle werden streng sequentiell pro Repository verarbeitet.



Damit kann nicht gleichzeitig:





Agent A dieselbe Datei verändern,



Agent B einen alten Zustand überschreiben,



das Handy einen Commit bestätigen,



während noch ein anderer Lauf schreibt.







2. Persistence Queue



Sie transportiert bestätigte Ergebnisse langsam zum NAS:



001 Commit-Objekte übertragen

002 neue Vault-Chunks übertragen

003 Evidence-Paket übertragen

004 Branch-Ref aktualisieren

005 Zustand als vollständig markieren





Diese Queue kann mit niedriger Priorität laufen:





nur ein Transfer gleichzeitig



Bandbreitenlimit



pausierbar



nach Verbindungsabbruch fortsetzbar



nachts schneller



während aktiver Nutzung gedrosselt











CUE wird das unveränderliche Kassenbuch



Nicht nur ein Logtext wie:



Upload abgeschlossen





Sondern ein strukturiertes Ereignis:



{

  "eventId": "evt-0184-009",

  "type": "state.accepted",

  "repository": "MYTHIC",

  "session": "run-0184",

  "baseCommit": "91af2c",

  "resultCommit": "b741d9",

  "intentId": "intent-oauth-overlay",

  "actor": "user",

  "evidence": [

    "diff:sha256:...",

    "preview:sha256:...",

    "flow-validation:sha256:..."

  ],

  "timestamp": "2026-07-14T20:18:00+02:00"

}





Danach:



{

  "type": "replication.queued",

  "commit": "b741d9",

  "target": "nas-primary"

}





Später:



{

  "type": "replication.verified",

  "commit": "b741d9",

  "target": "nas-primary",

  "checksum": "sha256:..."

}





CUE unterscheidet dadurch sauber:



lokal gespeichert

≠

auf NAS übertragen

≠

auf NAS verifiziert

≠

extern veröffentlicht





Das ist wichtig. Sonst meldet das System „fertig“, obwohl nur der lokale Rechner den Zustand besitzt.







Der Commit darf nicht auf das NAS warten



Ein schlechter Ablauf wäre:



Commit geklickt

→ 4 GB zum NAS übertragen

→ Nutzer muss warten

→ erst danach weiterarbeiten





Richtig:



Commit geklickt

→ lokal sofort abgeschlossen

→ CUE: Stable, lokal gesichert

→ NAS-Replikation läuft unabhängig weiter





Die Oberfläche zeigt beispielsweise:



MYTHIC · b741d9



Lokal:        ✓ gesichert

NAS:          ↑ 38 %

GitHub:       nicht veröffentlicht

Evidence:     ✓ vollständig





Du kannst währenddessen längst am nächsten Zustand arbeiten.







Änderungen müssen eine Basisversion tragen



Jeder Handyauftrag enthält den Commit, auf den er sich bezieht:



{

  "baseCommit": "91af2c",

  "command": "change_oauth_flow"

}





Wenn sich das Repo inzwischen weiterentwickelt hat:



Auftrag basiert auf: 91af2c

Aktueller Zustand:   b741d9





darf ANVIL nicht still überschreiben.



Es versucht:





Änderung auf neuen Zustand übertragen.



Prüfen, ob dieselben Bereiche verändert wurden.



Bei sauberer Übertragung fortfahren.



Bei echtem Konflikt Act Now anzeigen.







Damit bleibt die Queue sequentiell, ohne alte Aufträge blind wegzuwerfen.







Große Dateien: Chunks statt Komplettupload



Neue oder veränderte große Dateien werden lokal in Blöcke zerlegt:



castle.glb

├── Chunk A

├── Chunk B

├── Chunk C

└── Chunk D





Auf dem NAS liegen A, B und D vielleicht schon. Dann wird nur C übertragen.



Jeder Chunk erhält:





Inhalts-Hash



Größe



Transferstatus



Prüfsumme



Referenzen auf Zustände







Bei WLAN-Abbruch wird nur der fehlende Teil fortgesetzt.



42 von 64 Chunks vorhanden

→ weiter mit Chunk 43





Nicht wieder bei null anfangen.







Temporär bedeutet nicht unsicher



Die Sandbox kann temporär sein, während ihre Änderungen trotzdem sicher aufgezeichnet werden.



Vor Agentenlauf



Checkpoint A





Nach Agentenlauf



temporäres Delta B





Nach deiner Freigabe



Commit B





Nach NAS-Verifikation



Commit B besitzt replizierte Sicherung





Beim Absturz kann ANVIL aus dem lokalen Journal rekonstruieren:





welcher Auftrag aktiv war



welcher Commit die Basis war



welche Dateien verändert wurden



welche Transfers abgeschlossen waren



welche Evidence noch fehlte











Die NAS-Reihenfolge muss atomar sein



Beim NAS darf niemals zuerst der sichtbare Branch-Zeiger aktualisiert werden und danach erst die benötigten Inhalte kommen.



Richtige Reihenfolge:



1. fehlende Git-Objekte übertragen

2. fehlende Vault-Objekte übertragen

3. Evidence übertragen

4. Prüfsummen kontrollieren

5. erst jetzt Branch-/Zustandsreferenz aktualisieren





So verweist das NAS nie auf einen halbfertigen Zustand.



Metapher:





Erst alle Möbel ins Haus bringen, dann die Adresse als bezugsfertig veröffentlichen.
